# Gronthee - Book Cataloging System Requirements

## Project Overview
Gronthee is a web-based application designed to automate the process of cataloging books by scanning their covers and title pages using AI. It leverages third party AI model (configurable) to extract detailed metadata from images and provides a user-friendly interface for reviewing, editing, and exporting the data.

## Core Features

### 0. Authentication & Multi-User Data Isolation
- **Magic-link sign-in**: Users sign in with their email via Supabase Auth (passwordless magic link). The unauthenticated app shows a sign-in screen; once signed in the user is taken to the Scan page. `useAuth()` exposes the current user and `signOut()`.
- **Auth callback**: After clicking the magic link, the user lands on `/auth/callback`, which exchanges the URL code for a Supabase session and redirects into the app. Errors during exchange are surfaced inline on the callback page.
- **Profiles table**: A `profiles` row (`{ id, username }`, FK to `auth.users`) is created the first time a user picks a username. The username dialog only shows when no profile row exists for the signed-in user.
- **Multi-user data isolation**: All user-owned tables — `profiles`, `books`, `sessions`, `export_configs`, `user_preferences` — have RLS enabled with policies that restrict reads/writes to rows where `user_id = auth.uid()` (or `id = auth.uid()` for `profiles`). One Supabase project safely hosts many users.
- **Server-side secret handling**: AI provider keys (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GEMINI_API_KEY`, `OPENROUTER_API_KEY`) and Cloudflare R2 credentials (`CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_R2_ACCESS_KEY_ID`, `CLOUDFLARE_R2_SECRET_KEY`, `CLOUDFLARE_R2_BUCKET_NAME`, `CLOUDFLARE_R2_PUBLIC_URL`) live exclusively in Vercel server-side env vars (no `VITE_` prefix). The browser only ever sees `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`. All write paths to AI providers go through `/api/scan`; all R2 uploads go through `/api/r2/presign`. Both endpoints verify the caller's Supabase JWT before doing any privileged work.
- **Sign out and reset**: "Sign out" ends the Supabase session. "Reset all data" deletes the user's `export_configs` and `user_preferences` rows, clears local UI cache, and signs out (books and sessions are kept in Supabase as a soft reset — users can restore them by signing back in).

### 1. Book Scanning & Data Extraction
- **Image Input**: Supports both file uploads (drag-and-drop) and real-time webcam capture. Accepted formats: JPEG, PNG, WEBP. Maximum file size: 25 MB per image, up to 4 images per scan. The webcam flow requires a secure context (HTTPS or localhost) — the dev server runs over HTTPS via `vite-plugin-mkcert` (locally-trusted certificate). It requests browser camera permission on activation; shows a loading state while prompting, the live feed once granted, and a specific error for each failure case: permission denied (Retry), no camera found, or camera in use (Retry). On mobile phones (Android Mobile / iPhone / iPod), the back (environment-facing) camera is used by default to facilitate book scanning; on desktops and tablets the front camera is used. When the camera feed becomes ready, the page auto-scrolls so the feed and capture button are fully visible (pushing the page header off screen). The video feed is size-constrained (`max-h-[52vh] w-auto`) to fit together with the capture button in the viewport; the container wraps the video tightly (`w-fit`) to avoid black side bars.
- **AI Analysis**: Uses third party AI models to analyze one or multiple images of a book. It should support multiple providers (i.e Claude, GPT, Gemini, OpenRouter). **API keys never reach the browser** — all AI calls go through a Vercel Edge Function (`/api/scan`) that reads keys from server-side environment variables (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GEMINI_API_KEY`, `OPENROUTER_API_KEY`, no `VITE_` prefix). The browser authenticates requests with the Supabase JWT (`Authorization: Bearer <token>`). The Edge Function streams the AI response as SSE; the client collects all chunks before parsing. Users get to choose a model from a dropdown list before scanning. OpenRouter models (e.g. Qwen 3.6 Plus, model ID `qwen/qwen3.6-plus:free`) are accessed via the OpenAI-compatible API at `https://openrouter.ai/api/v1`. Gemma 4 (model ID `gemma-4-31b-it`) is available under the Google Gemini provider and is accessed via the `@google/generative-ai` SDK.
- **English Language Emphasis**: All scan results (author, genre, summary, publisher, etc.) are provided in English or romanised/transliterated form, even if the original book is in a non-English language. **Exception — Book Title**: the `title` field always uses the title exactly as it appears on the book (romanised if non-Latin script). If an English translation of the title also appears on the book, it goes into `otherTitle`. An English translation must never be placed in `title` when the original title is available.
- **Metadata Extraction**: Automatically identifies and extracts the following fields:
    - **Title**: Primary title of the book.
    - **Sub Title**: Secondary title or subtitle.
    - **Other Title**: Additional titles (defaults to empty).
    - **Author**: Primary author's name. If the AI returns no value, defaults to `N/A`.
    - **Second Author**: Secondary author(s), if applicable.
    - **Editor**: Editor's name (common for collections).
    - **Translator**: Name of the translator for translated works.
    - **Illustrator**: Name of the illustrator.
    - **Publisher**: Publishing house name. For Bengali books, the AI is instructed to use the name listed under প্রকাশক (publisher) — not পরিবেশক (distributor). If both appear, প্রকাশক always takes priority.
    - **Published Year**: Year of publication in Gregorian YYYY format. When multiple editions with different years appear on the book, the AI always picks the latest year. The AI also extracts or derives the Bengali era (Bangla Saal) year (`publishedYearBengali`). If only a Bengali year appears on the book (e.g. "Boishakh 1407"), it is converted to Gregorian by adding 593. If only a Gregorian year is present, the Bengali year is derived by subtracting 593. Both fields are always filled when at least one is determinable. In the editor, when a Bengali year is present, a note is shown below the Year field: "Bengali calendar: {year} BS".
    - **ISBN**: International Standard Book Number. Dashes must be stripped on save (e.g. `978-3-16-148410-0` → `9783161484100`). If the AI returns no value, defaults to `N/A`.
    - **Category**: High-level classification. Dropdown with fixed options: "Fiction", "Non-Fiction", "Miscellaneous". AI is instructed to return one of these exact values. If the AI returns a value not in the list, it is reset to empty and the user must select manually.
    - **Genre**: Specific genre. Dropdown with fixed options: Agriculture, Analytical Essays, Art, Art History, Autobiography, Belles Letters, Bilingual, Biography, Children, Collected Works, Comedy, Cooking, Crime, Culture, Dictionary, Drama, Education, Essay, Fiction, Historical Fiction, History, Indian Magazine, Indian Philosophy, Islamic History, Letters, Literary Criticism, Lyrics, Magazine, Math, Memoir, Miscellaneous, Muktijuddha, Mythology, Nature, Non-Fiction, Novel, Novella, Partitition, Philosophy, Play, Poetry, Reference, Religion, Sanskrit, Science, Science Fiction, Short Stories, Song, Spirituality, Story, Tagore, Thriller, Translation, Travel, Travelogue, Workbook.
    - **Collection**: A coded field mapped to specific categories (see Mappings below).
    - **Item Type**: A coded field representing the physical type of the item (see Mappings below).
    - **Page Count**: Total number of pages.
    - **Language**: ISO 639-1 language code in ALL CAPS (e.g., EN, BN, FR).
    - **Edition**: Ordinal edition number (e.g., 1st, 2nd, 3rd). Only edition or publication dates are considered — impression numbers, print run numbers, and reprint dates are ignored. If multiple editions are mentioned, always use the highest (latest) one.
    - **Publication Place**: City of publication only (no country or state).
    - **Scan Date**: Automatically generated current date in YYYY-MM-DD format (Non-editable).
    - **Summary**: A brief 1-2 sentence summary of the book.

### 2. Data Management
- **Edit Form**: A comprehensive form allowing users to review and correct AI-extracted data before saving. Most fields are mandatory — saving is blocked if any required field is empty, with inline error messages and red border highlights on each invalid field. Errors clear as the user fills in each field. When save fails due to validation, a red error banner is shown below the form fields instructing the user to fill in all highlighted fields; the banner disappears on a successful save. Optional fields (no validation) are: Subtitle, Other Title, Second Author, Editor, Translator, Illustrator. Required fields that are empty on initial load (e.g. when AI could not extract a value) show an amber background (`bg-amber-50`) immediately, before any save attempt, to prompt the user to fill them in.
- **User Preference Storage**: The system learns from user edits to the **Author** field only. If the user manually corrects the author name and the edit is a minor correction (Levenshtein distance ≤ 30% of the original length, e.g. max 3 characters changed out of 10), the mapping is stored in Supabase (`user_preferences.author_mappings`, RLS-scoped to the authenticated user) so it persists cross-device. Future scans that produce the same original author value will automatically be corrected to the user's preferred value. Edits that exceed the threshold (e.g. completely rewriting the name) are not stored.
- **Display View**: A clean, structured view of the extracted metadata.
- **History Tracking**: Saved books are persisted in Supabase (`public.books`, RLS-scoped to the authenticated user). Each `BookMetadata` record carries a `user_id` so different users never see one another's catalogues. On first login after upgrading from a localStorage-only build, `LocalStorageMigrationDialog` imports any existing `gronthee:books` and `gronthee:sessions` data into Supabase and sets `gronthee:migrated` so it runs only once.
- **Image Storage (Cloudflare R2)**: Scanned images are compressed (max 1200 px, JPEG quality 0.8) and uploaded to Cloudflare R2 when the user saves a new book. Files are stored at `{bookId}/{index}.jpg` (1-based) in the configured R2 bucket and are publicly accessible via the R2 public URL. The resulting URLs are stored in `BookMetadata.imageUrls`. Upload happens before `onAdd` is called so URLs are available immediately in the saved record. If upload fails (partially or fully), a non-blocking amber warning is shown and the book is saved without image URLs. R2 credentials live server-side only — the browser calls `/api/r2/presign` (a Vercel Function authenticated with the user's Supabase JWT) to obtain short-lived (5 min) presigned `PUT` URLs plus the corresponding public URLs, then uploads each compressed blob directly to R2. Server env vars: `CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_R2_ACCESS_KEY_ID`, `CLOUDFLARE_R2_SECRET_KEY`, `CLOUDFLARE_R2_BUCKET_NAME` (optionally `bucket/prefix`), `CLOUDFLARE_R2_PUBLIC_URL`. No R2 credentials reach the browser. Image URLs are not part of any Export Config column schema (R&D-only fields are excluded from `BookField`); they are accessible via the optional R&D columns checkbox at export time. Book deletion does not currently remove objects from R2.
- **CSV Export**: Export is per-session and config-driven. Clicking "Export CSV" on the History page opens `ExportSessionDialog`, which lists every session. Each row shows the session name, the name of its Export Config as a subtitle, and a book count, alongside a dedicated **Export** button. Clicking a row's Export button immediately downloads that session's CSV — named `gronthee-<username>-<sessionSlug>-<datetime>.csv` — using the column schema defined by `getConfig(session.configId)`. The dialog stays open so multiple sessions can be exported one at a time. There are no checkboxes or bulk-select controls; a single Cancel button dismisses the dialog. Pre-existing sessions use the built-in **Dishari** config, which reproduces the original CSV column order byte-for-byte (backward compatible). The `Collection` and `Item Type` columns in the Dishari config export abbreviated codes (e.g. `FIC`, `BK`), not human-readable labels; custom configs may export whichever fields and headers the user configures.
  - **R&D columns (optional)**: The export dialog includes an "Include R&D columns" checkbox (unchecked by default; resets each time the dialog opens). When checked, three additional columns are appended after whatever columns the session's config produces: `Book ID` (the internal book UUID), `Images` (JSON-encoded array of public R2 URLs, e.g. `["…/1.jpg","…/2.jpg"]`, empty string for books with no uploaded images), and `Prompt Output` (the JSON body returned by the AI for that scan, with any surrounding markdown fences or prose stripped — stored as a string starting with `{` and ending with `}`; empty for pre-feature books or books saved without a scan). The downloaded filename is prefixed with `gronthee-rnd-` in this mode: `gronthee-rnd-<username>-<sessionSlug>-<datetime>.csv`. R&D columns are appended regardless of which Export Config the session uses, so output is consistent across all sessions. These columns are intended for research and data-quality analysis and are not part of any config definition. The raw AI output is captured at scan time and persisted on the `BookMetadata` record under `rawAIOutput`.
- **Username**: After signing in for the first time, the app shows a welcome popup asking the user to choose a username. The username is stored in Supabase under `profiles.username` (one row per `auth.users.id`, RLS-scoped) and persists across devices. The popup is only shown once per account (when no `profiles` row exists yet). The username is displayed as a greeting ("Welcome, \<username\>") above the heading on the Scan page, and is used to personalise the default CSV export filename. Whitespace characters are not allowed in the username — they are stripped from the input as the user types.
- **Page Count Prompt**: When the user clicks "Scan Book", a popup dialog is shown immediately — before the AI call is made — asking for the number of pages. The user may enter a value or skip. The value is stored and merged into the scanned metadata after the AI call completes, before navigating to the editor. On mobile, the dialog adjusts its position using the `visualViewport` API so it is not obscured by the on-screen keyboard.
- **Sessions**: Users can organise scans into named sessions. Each saved book belongs to the currently active session. Sessions are managed via a `SessionSelector` dropdown shown on the Scan page (below the heading) and implicitly assigned when saving a book from the Editor page. On the Review & Save page (new book only), the active session is shown below the Save button as a right-aligned label — "Active session: `<name>`" — so users know which session the book will be saved to before confirming.
  - **Session data model**: A `Session` has `{ id, name, createdAt, configId }`. The `id` is a URL-safe slug derived from the name. `configId` is the id of the export config used when this session is exported to CSV (defaults to `'dishari'`). Every `BookMetadata` record has a `sessionId` field.
  - **Default session**: A built-in "Default" session always exists (`id = 'default'`, `configId = 'dishari'`). It cannot be renamed or deleted. Existing books that predate sessions are automatically migrated to `sessionId = 'default'` on first load.
  - **Creating sessions**: Users create new sessions through a small **New Session** dialog opened from the `SessionSelector` dropdown. The dialog has a name field and an Export Config dropdown (defaulting to Dishari). Names must be unique (case-insensitive). Maximum 50 sessions total. Submitting the dialog also makes the new session the active session.
  - **Session subtitle**: Each row in the `SessionSelector` shows the associated config name as a small subtitle so users can see at a glance which schema each session uses. The active session's config name is also shown inline on the Scan page header (next to the session name, separated by a `·` bullet) so users can confirm the export schema before scanning.
  - **Renaming sessions**: Any session except Default can be renamed inline via a pencil icon. The icon is always visible on touch devices and hover-revealed on desktop (≥ `sm` breakpoint). Name length capped at 50 characters.
  - **Deleting sessions**: Any session except Default can be deleted via a trash icon (same touch-vs-desktop visibility rule as the rename icon). A confirmation dialog shows the number of books that will be moved to Default before proceeding. Books are never orphaned — they are always reassigned to Default on session delete.
  - **Switching sessions**: Selecting a session from `SessionSelector` updates the active session for future scans. The last-used session is persisted under `gronthee:currentSessionId` and restored on next visit. If the stored session no longer exists, falls back to Default.
  - **History filtering**: The History page shows session filter chips (All + one per session). Clicking a chip filters the book list to that session only. Filter is local UI state — does not change the active scan session.
  - **Export per session**: The Export CSV dialog lists all sessions, each showing its name, the name of the export config it uses (as a small subtitle), and book count alongside a dedicated Export button. Clicking a row's Export button downloads that session's CSV immediately (named `gronthee-<username>-<sessionSlug>-<datetime>.csv`) using the schema defined by that session's config (resolved via `getConfig(session.configId)`), without closing the dialog, allowing multiple sessions to be downloaded one at a time. There are no checkboxes or bulk-select controls. The dialog has a single Cancel button at the bottom right to dismiss it.
  - **Persistence**: Sessions are stored in Supabase `public.sessions` for authenticated users. `gronthee:currentSessionId` (active session id) remains in localStorage as UI state only. On first login, existing `gronthee:sessions` localStorage data is migrated to Supabase via `LocalStorageMigrationDialog`. Both keys are cleared on "Reset all data".
- **Export Configs**: Users can define their own CSV export schemas as named configs and choose which one each session uses. Configs are managed on a top-level **Configs** page reachable from the header navigation alongside Scan and History.
  - **Config data model**: An `ExportConfig` has `{ id, name, description?, builtIn?, columns[], createdAt, updatedAt }`. Each `ExportColumn` is either `Mapped` (`{ type: 'mapped', header, field }` — pulls a value from a `BookMetadata` field) or `Constant` (`{ type: 'constant', header, value }` — always emits a fixed string). The set of bindable fields excludes R&D-only fields (`id`, `imageUrls`, `rawAIOutput`); those are surfaced exclusively through the R&D columns checkbox at export time.
  - **Built-in Dishari config**: The bundled config at `src/config/dishari-export-config.json` (id `dishari`) reproduces the existing CSV shape used by the external library system. It is read-only — Edit and Delete are hidden in the UI; users may Clone or Share it.
  - **Configs page**: Lists every config (built-in + user) as cards with name, optional description, column count, and a Built-in badge for Dishari. Each card surfaces actions: **Edit** (user configs only), **Clone** (creates a new editable user config seeded with the source's columns), **Share** (downloads the config as JSON), **Delete** (user configs only). The page header has **New Config** (opens an empty editor) and **Import** (opens an import dialog).
  - **Editor**: Full-page editor for create and edit. Fields: name (required, unique case-insensitive, max 80 chars), optional description (max 200 chars), and an ordered column list. Each row exposes a header text input, a Mapped/Constant type select, and either a field dropdown or a value input. Rows can be reordered via drag-and-drop or keyboard up/down arrow buttons; rows can be removed individually; an "Add column" button appends a new row. Save validates and persists; Cancel returns to the list without saving.
  - **Sharing**: The Share action serialises the config to a pretty-printed JSON file named `gronthee-config-<slug>.json`. The shared payload omits the internal id and `builtIn` flag — the recipient re-derives a slug on import and the imported config is always editable.
  - **Importing**: The Import dialog accepts a JSON file, validates it (rejects missing name, empty/missing columns, unknown column type, mapped columns referencing fields outside the supported set, headers longer than 200 chars, more than 200 columns), shows a preview with an editable name to resolve naming collisions, and adds it to the user's configs on confirm.
  - **Deleting a config**: Removing a user config is destructive. Before deletion, any sessions referencing that config are reassigned to Dishari (`configId = 'dishari'`) and the change is persisted; `getConfig` also falls back to Dishari for unknown ids as a defence-in-depth safeguard. A confirmation dialog calls this out before deleting.
  - **Export wiring**: When a session is exported, its CSV is generated from the schema defined by `getConfig(session.configId)`. The `ExportSessionDialog` shows each session's resolved config name as a subtitle so users can see at a glance which schema each session will use before clicking Export. The R&D-columns checkbox remains orthogonal and continues to append `Book ID, Images, Prompt Output` (and apply the `gronthee-rnd-` filename prefix) regardless of which config the session is on.
  - **Persistence**: User-created configs are stored in Supabase (`export_configs` table, RLS-scoped to the authenticated user); the Dishari built-in is never persisted to DB. On "Reset all data", the user's `export_configs` rows are deleted from Supabase.
- **User Preferences (author corrections)**: Author name corrections recorded by the user are stored in Supabase (`user_preferences.author_mappings`), persisting cross-device. Applied automatically on next scan.
- **Supabase Free-Tier Keepalive**: A daily Vercel cron job (`/api/cron/keepalive`, scheduled `0 12 * * *`) issues a lightweight `SELECT` against the `profiles` table to prevent the free-tier Supabase project from auto-pausing after 7 days of inactivity. The endpoint requires `Authorization: Bearer ${CRON_SECRET}` (Vercel attaches this automatically to scheduled invocations) so the public URL cannot be triggered by anyone else. Returns JSON `{ ok: true }` on success.
- **Reset All Data**: A "Reset all data" link at the bottom of every page clears all app data. Clicking it shows a confirmation dialog. Confirming deletes the user's `export_configs` and `user_preferences` Supabase rows, clears remaining localStorage keys (`gronthee:currentSessionId`, `gronthee:selectedModel`, `gronthee:migrated`), and signs out. Books and sessions remain in Supabase (reset is a sign-out + local-cache clear, not a full data purge). The button is styled in red to reflect its destructive nature.
- **Persistent Model Selection**: The user's selected AI provider and model are persisted in `localStorage` under the key `gronthee:selectedModel` (stored as `{provider, modelId}` — the API key is never stored). On next visit, the last-used model is restored automatically. Falls back to the configured default if the saved value is missing or invalid.
- **Persistent Input Source (Webcam Memory)**: If the user selects the Webcam tab, that choice is saved in `localStorage` under the key `scanner-tab`. When the user returns to the Scan page the webcam feed starts immediately. If there is no user activity (mouse move, click, keydown, touch, scroll) for 3 minutes while on the Webcam tab, the app automatically switches back to the Upload tab and removes the saved preference — prompting the user to actively re-select webcam next time.

### 3. User Interface
- **Branding**: The browser tab shows a book icon favicon (SVG, indigo color scheme matching the app) and the page title "Gronthee".
- **Responsive Design**: Optimized for both desktop and mobile usage.
- **Modern Aesthetic**: Built with Tailwind CSS, featuring a clean, professional look with smooth animations (using Motion).
- **Interactive Feedback**: Loading states, success indicators, and error handling for a seamless experience.
- **Mobile Image Preview**: On the Review & Save page, scanned images are displayed in a 2-column grid on mobile (single column on desktop/tablet). The thumbnail grid is constrained to `max-w-[240px]` on mobile so images remain compact (~120px each). Each thumbnail uses a 3:4 aspect ratio. Tapping an image opens a full-screen zoom modal with:
  - Zoom steps: 1×, 1.5×, 2×, 3× — controlled via bottom toolbar (ZoomIn/ZoomOut buttons + dot indicators)
  - Tapping the image itself toggles between 1× and 2×
  - Current zoom level shown as a badge in the top-left
  - At zoom > 1× the image is scrollable within the modal
  - Close button (top-right) or tapping the backdrop dismisses the modal and resets zoom to 1×

## Technical Stack
- **Frontend**: React (TypeScript)
- **Styling**: Tailwind CSS
- **Icons**: Lucide React
- **Animations**: Motion (motion/react)
- **AI Integration**: Multiple third party providers; API keys stored as Vercel environment variables
- **Camera**: react-webcam
- **File Handling**: react-dropzone
